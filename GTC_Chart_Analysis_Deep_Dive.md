# GTC 2026 图表深度解析：Groq LPU 的技术诱惑与架构挑战

## 执行摘要

本文基于 NVIDIA GTC 2026 主题演讲中的关键图表，深度解析 Groq LPU（Language Processing Unit）的技术架构、商业逻辑、以及在实际部署中面临的三大挑战。LPU 代表了 AI 推理硬件从通用化到专用化的根本性转变，但其独特的架构设计也带来了前所未有的软件和系统级挑战。

## 1. GTC 2026 关键图表：Pareto 前沿的重新定义

### 1.1 图表核心信息

**纵轴**：吞吐效率（TPS/MW - Tokens Per Second per Megawatt）
**横轴**：交互性（TPS/User - Tokens Per Second per User）

图表将 Token 生成分为四个市场层级：
- **Free 层**：Qwen 225B，50 TPS，$0
- **Medium 层**：Kimi 1T，100 TPS，$3  
- **High 层**：GPT 2T，200 TPS，$6
- **Premium 层**：GPT 2T，400 TPS，$45

### 1.2 关键洞察

#### 1.2.1 LPX 的价值定位
- **左边市场（50-100 TPS）**：NVIDIA Rubin NVL72 独立即可满足
- **右边市场（200-400 TPS）**：Rubin 单独吞吐急剧下降，而 **Rubin + LPX 组合维持了高性能**
- **性能对比**：在 400 TPS 下，Rubin + LPX 比 Blackwell 高 **35 倍** 的吞吐效率

#### 1.2.2 图表隐藏的两个关键细节

**细节一：Rubin + LPX 曲线几乎是平的**
- 从 200 TPS 到 400 TPS，吞吐几乎没有下降
- 对比：Rubin NVL72（亮绿线）随着 TPS/User 提高急剧下降
- **技术原因**：GPU 在高 TPS 要求小 batch，小 batch 算力利用率低；LPU 的算力带宽比只有 4，从一开始就没吃到大 batch 的算力收益

**细节二：右半边是 2T 模型，LPU 的容量撑不住**
- High/Premium 档对应的是 **GPT MoE 2T 模型**
- 即使 W4 量化也要 1 TB
- LPX 整个机柜的 SRAM 只有 128 GB（256 颗 LPU × 500MB），连 1/8 都放不下

### 1.3 根本性权衡
**LPU 用算力和容量换来了极致带宽**：
- 35 倍的 TPS/MW 提升背后
- 算力天花板（平线）
- 容量缺口（装不下 2T 模型）

## 2. Groq LPU：确定性执行的 SRAM 推理芯片

### 2.1 商业背景：产能是第一性

#### 2.1.1 交易结构
- **时间**：2025 年底
- **金额**：200 亿美元
- **结构**：IP license + hiring（非法律意义上的收购）
- **战略考量**：避免反垄断审查，快速获得 IP 和人才

#### 2.1.2 产能战略
- **工艺**：三星 SF4（4nm）
- **优势**：不占台积电 N3 产能，不需要 HBM，不需要 CoWoS
- **结果**：英伟达在不牺牲 GPU 产能的情况下，开辟了一条完全独立的增量产线

### 2.2 LP30 芯片架构：巨大的带宽，巨大的牺牲

#### 2.2.1 架构对比

| 指标 | LP30（单卡） | Rubin Ultra GPU（单卡） | 对比 |
|------|-------------|----------------------|------|
| **容量** | 500 MB SRAM（片上） | 288 GB HBM4（片外） | 1:576 |
| **带宽** | 150 TB/s | 22 TB/s | 6.8:1 |
| **FP8 算力（dense）** | 1.2 PFLOPS | ~15.75 PFLOPS | 1:13 |
| **算力带宽比** | **4** | **~358** | 1:90 |
| **工艺/封装** | 三星 SF4，单 die，无先进封装 | 台积电 N3P + CoWoS + HBM4 | - |
| **互连** | 96 × 112G SerDes C2C | NVLink，1.8 TB/s | - |

#### 2.2.2 架构哲学
- **核心设计**：抛弃多级缓存 + 动态调度，用单层 SRAM + 编译器静态调度实现确定性执行
- **功能单元**：MXM（矩阵乘法）、VXM（向量运算）、MEM（内存读写）、SXM（数据搬运/转置）
- **数据流**：东西向流过流水线，南北向做 SIMD 并行

#### 2.2.3 算力带宽比 = 4 的含义
- **含义**：batch size 超过个位数就进入 Compute-bound，SRAM 带宽优势消失
- **设计目标**：只做一件事——在 batch=1 以极低延迟完成内存密集型运算
- **结果**：第一章那条平线的根源就在这里

### 2.3 集群及互联架构：四级带宽衰减

#### 2.3.1 ComputeTray（1U 计算托盘）
- **组成**：8 颗 LP30 + 2 颗 Altera FPGA（FEL）+ Intel Granite Rapids CPU + BlueField-4 网卡
- **规格**：SRAM 合计 4 GB / 1.2 PB/s，FP8 算力 9.6 PFLOPS

#### 2.3.2 Fabric Expansion Logic（FEL）
- **角色**：LPU C2C 和外部以太网之间的协议桥接 + SRAM 放不下的专家参数暂存
- **技术**：2 颗 FPGA 挂载 256GB DDR5
- **现状**：time-to-market 的妥协，当前系统最大的瓶颈之一

#### 2.3.3 互连拓扑与带宽衰减

| 去向 | Link 数 | 带宽 | 特点 |
|------|---------|------|------|
| **Tray 内全连接** | 56 | 5.6 Tbps | C2C 直连，确定性，延迟最低 |
| **Tray 间（机柜内）** | 31 | 3.1 Tbps | 背板走线，仍然是 C2C |
| **机柜间** | 4 | 400 Gbps | **前面板出去 → 以太网**，延迟和抖动显著增大 |
| **FEL FPGA** | 4 | 400 Gbps | 参数加载 + 以太网桥接 |

**关键洞察**：带宽从内到外衰减 **14 倍**（5.6T → 400G）。GPU ↔ LPU 的通信恰好走最外层的以太网——这个带宽层级直接决定了 AFD 的通信开销。

### 2.4 确定性执行：现代芯片复杂性的根本重构

#### 2.4.1 传统芯片的"不确定性兜底"
现代芯片架构的大部分复杂性，都是在兜住不确定性：
- **多级缓存**：赌数据在不在附近
- **乱序执行**：填住等待的空隙
- **重排序缓冲区、指令队列、Store Buffer**：在不确定的延迟下维持吞吐
- **网络层面的拥塞控制、动态路由、仲裁器**：应对数据包到达时间和路径的不确定性

#### 2.4.2 Groq 的逆向思维：从源头消除不确定性
- **单层 Scratchpad SRAM**：没有缓存层次，不存在 cache miss
- **编译器静态调度**：编译时决定每个时钟周期的数据位置和指令
- **确定性片间网络**：编译时规划通信路径和时序，没有运行时争用
- **硬件 deskew 指令**：周期性校正多芯片时钟漂移，维持全局同步

#### 2.4.3 面积重新分配
**省下来的面积全部填了 SRAM**：
- 不需要 L1/L2/L3 缓存
- 不需要乱序执行单元
- 不需要重排序缓冲区
- 不需要指令队列
- 不需要拥塞控制硬件

**结果**：LP30 能在一颗 die 上塞 500MB、做到 150TB/s 带宽

#### 2.4.4 代价：灵活性
- 模型架构变了就要重新编译
- 大模型架构迭代速度极快
- LPU 的先编译架构能否跟上 TensorRT-LLM 的适配节奏，决定了 400-500 万颗 LPU 是"卖出去就能用"，还是"卖出去了但用不起来"

## 3. AFD 分离：架构创新与三大挑战

### 3.1 从 PD 分离到 AFD 分离

#### 3.1.1 PD 分离（Prefill-Decode 分离）
- **原理**：Prefill（上下文编码）和 Decode（自回归生成）对延迟的敏感度不同
- **部署**：Prefill 用 GPU（高吞吐），Decode 用 LPU（低延迟）
- **基础**：SLA（服务等级协议）差异

#### 3.1.2 AFD 分离（Attention-FFN-Decode 分离）
- **原理**：将 Decode 进一步拆分为 Attention 和 FFN（前馈网络）
- **部署**：
  - **Attention**：GPU（容量大，算力高）
  - **FFN**：LPU（带宽高，延迟低）
  - **Decode**：协调层
- **基础**：必须靠异构硬件

### 3.2 挑战一：容量——SRAM 放不下万亿模型

#### 3.2.1 容量缺口
- **GPT MoE 2T 模型**：W4 量化后约 1 TB
- **LPX 机柜 SRAM**：128 GB（256 颗 × 500MB）
- **缺口**：仅能容纳 1/8 的模型参数

#### 3.2.2 FEL DDR5 的局限性
- **容量**：256 GB（2 颗 FPGA × 128GB）
- **带宽**：~400 GB/s（DDR5-6400 × 8 channels）
- **对比**：比 SRAM 慢 375 倍（150 TB/s vs 400 GB/s）
- **结果**：从 FEL 加载参数到 SRAM 成为瓶颈

### 3.3 挑战二：通信——每层两次跨机柜以太网

#### 3.3.1 AFD 通信模式
```
GPU（Attention） → LPU（FFN） → GPU（Attention） → ...
```
- **每层**：2 次跨机柜通信
- **每次通信**：7 跳（GPU HBM → GPU 内存控制器 → PCIe → CPU → 以太网卡 → 交换机 → LPU FEL → LPU SRAM）

#### 3.3.2 通信开销分析
- **单次通信延迟**：~10 μs（理想）到 ~100 μs（实际）
- **每层总延迟**：20-200 μs
- **占比**：在 400 TPS（2.5 ms/token）场景下，通信可能吃掉 **40%** 的耗时

#### 3.3.3 带宽瓶颈
- **机柜间带宽**：400 Gbps（50 GB/s）
- **对比 SRAM 带宽**：1/3000（150 TB/s vs 50 GB/s）
- **结果**：通信成为系统瓶颈

### 3.4 挑战三：调度——BSP 失效带来的颠覆式需求

#### 3.4.1 BSP（Bulk Synchronous Parallel）模型的失效
- **传统模式**：所有芯片完成当前层 → 全局同步 → 进入下一层
- **AFD 问题**：GPU 和 LPU 计算时间不同，强制同步造成资源闲置

#### 3.4.2 全异步调度的需求
- **目标**：让 GPU 和 LPU 各自以最快速度推进，通过流水线隐藏通信延迟
- **挑战**：需要全新的调度框架和编译器支持

## 4. 解法与展望：全异步调度 + 硬件升级

### 4.1 软件解法：全异步调度框架

#### 4.1.1 核心思想
- **解耦**：GPU 和 LPU 不再强同步
- **流水线**：GPU 计算第 N 层 Attention 时，LPU 计算第 N-1 层 FFN
- **缓冲区**：中间结果缓冲，平滑计算和通信的波动

#### 4.1.2 技术挑战
- **依赖关系管理**：确保数据一致性
- **资源调度**：动态分配计算和通信资源
- **容错机制**：处理硬件故障和性能波动

### 4.2 硬件升级路径

#### 4.2.1 短期（2026-2027）：FEL ASIC 化
- **目标**：用定制 ASIC 替换 FPGA
- **收益**：降低延迟，提高能效，增加带宽

#### 4.2.2 中期（2027-2028）：C2C 直连 GPU-LPU
- **目标**：绕过以太网，实现 GPU 和 LPU 的直接高速互连
- **技术**：NVLink-C2C 桥接芯片
- **收益**：通信延迟降低 10 倍，带宽提高 10 倍

#### 4.2.3 长期（2028+）：统一内存架构
- **目标**：GPU HBM 和 LPU SRAM 共享统一地址空间
- **技术**：CXL 3.0+，内存语义网络
- **收益**：彻底消除数据拷贝，实现真正的异构计算

### 4.3 商业前景与风险

#### 4.3.1 市场机会
- **高端推理市场**：400+ TPS 的 Agent 场景
- **边缘推理**：确定性延迟的关键应用
- **混合云**：GPU+LPU 的异构部署

#### 4.3.2 技术风险
- **软件生态**：编译器、调度框架的成熟度
- **模型适配**：新模型架构的快速支持
- **系统集成**：大规模部署的可靠性

#### 4.3.3 竞争态势
- **AMD**：MI400 + XDNA3 的异构策略
- **Intel**：Gaudi 4 + 定制加速器
- **中国厂商**：华为 Ascend + 定制芯片

## 5. 结论：确定性计算的新范式

### 5.1 技术范式转移
Groq LPU 代表了一种根本性的技术范式转移：
- **从**：用复杂度兜住不确定性
- **到**：用确定性换掉复杂度

### 5.2 商业逻辑重构
- **第一性**：产能独立，不依赖台积电和内存厂
- **第二性**：用算力和容量换极致带宽
- **第三性**：确定性执行带来的服务质量保证

### 5.3 系统级挑战
AFD 分离暴露了三个核心挑战：
1. **容量挑战**：SRAM 容量 vs 模型规模
2. **通信挑战**：异构硬件间的数据移动
3. **调度挑战**：从同步到异步的范式转变

### 5.4 未来展望
LPU 的成功不仅取决于芯片本身，更取决于：
- **软件栈**：编译器、调度框架、运行时系统的成熟度
- **生态系统**：模型支持、工具链、开发者社区
- **系统集成**：与 GPU、CPU、网络、存储的协同

**最终判断**：LPU 不是 GPU 的替代品，而是 GPU 的补充。在 AI 推理的 Pareto 前沿上，GPU+LPU 的异构组合正在定义新的性能边界。成功的关键在于能否解决 AFD 分离带来的三大挑战，这需要芯片、软件、系统三个层面的协同创新。
